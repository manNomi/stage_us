#Server - Clinet 구조 
	- 사용자는 서버에 접속해서 코드를 받아간다 그 코드를 받아서 사용자가 보는게 기본적인 구조 
	- 그 코드를 눈으로 보기위해서 웹브라우저를 사용한다 
	- 네이버 .com 은 서버의 주소 / 브라우저는 서버를/  해석해 우리눈에 보여준다
	{사용자 -> 웹 브라우저 -> 서버}
	-> 왜 이 구조냐 : 개발자가 편하기 때문 
				-> 코드 파일은 서버에 있는데 서버만 코드를 바꾸면 사용자가 변해도 자동으로 가능 
	  웹브라우저를 거치기 때문에 개발자가 편하지만 느린구조다 
	  게임의 경우 1초뒤에 되거나 하면 매우 타격이 크다 
	  웹은 사용자가 기다려 줄 수 있기에 느려도 개발자가 편리한 서버 클라이언트 구조를 사용한다 
#프론트엔드 & 백엔드
	- 프론트엔드 : 눈에 보이는 공간 (로그인 창 등 버튼을 누르는 일까지)
		-> HTML (뼈대를 잡아준다,화면 구역) HTML5
		-> CSS (뼈대를 꾸며주는 , 색깔 등 ) CSS3
		-> SCRIPT (화면에 변화를 주는 , 버튼을 누르면 버튼 색 바꾸는 등) Javascript
	- 백엔드 : 눈에 보이지 않는 공간 (로그인 버튼을 눌렀을때 로그인을 검증해주는 역할)
		->SERVER (JSP)
-jsp
Java[1]를 이용한 서버 사이드 템플릿 엔진이다. jsp 는 자바를 웹서버에서 쉽게 쓰기 위한 기술이며 언어가 아니다.
 Java Server Pages의 약자이며, 오라클에서 자바상표권 문제로 오픈소스인 jsp는 자카르타 서버페이지로 이름을 바꾸었다. 
Java의 점유율을 대폭 상승시킨 1등공신이다. 같은 부류에 속하는 것으로 ASP가 있다. 확장자는 당연히 .jsp를 사용.
 ASP와 마찬가지로 <% ... %>로 둘러싸인 스크립트 영역이 있으며, 실행시에 javax.servlet.http.HttpServlet 클래스를 상속받은
 Java 소스 코드로 변환한 다음[2] 컴파일되어 서블릿으로 실행된다. 이 JSP 파일을 Servlet 자바소스로 클래스로 변환하고 컴파일 역할을 하는 프로그램을
 JSP 컨테이너라 하며 컴파일된 서블릿 클래스 파일을 실제로 실행하는 프로그램은 서블릿 컨테이너 혹은 웹 컨테이너[3]라고 부른다.[4][5]로 부른다. 
대표적인 것으로 Tomcat. 하나의 JSP 페이지가 하나의 Java 클래스이기 때문에 모든 Java 라이브러리를 끌어다 쓸 수 있다.
